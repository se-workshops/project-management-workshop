## 行動規範

このプロジェクトでは、すべての参加者が互いに尊重し合うことを期待しています。

- 建設的で前向きなフィードバックを提供する
- 異なる視点や経験を尊重する
- プロフェッショナルな態度を保つ

## 開発環境のセットアップ

### 前提条件

- Node.js (v18以上)
- npm または yarn

### セットアップ手順

1. リポジトリをフォークする
2. ローカルにクローンする

```bash
git clone https://github.com/YOUR_USERNAME/project-management-workshop.git
cd project-management-workshop
```

3. 依存関係をインストールする

```bash
npm install
```

4. 開発サーバーを起動する

```bash
npm run dev
```

フロントエンドは `http://localhost:5173` で起動します。

5. バックエンドサーバーを起動する（別のターミナルで）

```bash
npm run server
```

バックエンドは `http://localhost:3000` で起動します。

## 貢献の方法

### バグ報告

バグを見つけた場合は、以下の情報を含めてIssueを作成してください：

- バグの詳細な説明
- 再現手順
- 期待される動作
- 実際の動作
- スクリーンショット（該当する場合）
- 環境情報（OS、ブラウザ、Node.jsバージョンなど）

### 機能提案

新機能の提案は歓迎します。Issueを作成して以下を含めてください：

- 機能の説明
- ユースケース
- 期待される効果
- 実装のアイデア（あれば）

### コード貢献

1. 既存のIssueを確認するか、新しいIssueを作成する
2. 作業ブランチを作成する

```bash
git checkout -b feature/your-feature-name
# または
git checkout -b fix/bug-description
```

3. コードを変更する
4. テストを追加・更新する
5. 変更をコミットする
6. プルリクエストを作成する

## コーディング規約

### JavaScript/React

- **フォーマット**: Prettierを使用（設定は`.prettierrc`参照）
- **リント**: ESLintを使用（設定は`.eslintrc`参照）
- **命名規則**:
  - コンポーネント: PascalCase（例: `ProductCard`）
  - 関数・変数: camelCase（例: `getUserData`）
  - 定数: UPPER_SNAKE_CASE（例: `API_BASE_URL`）
  - ファイル名: コンポーネントはPascalCase、その他はcamelCase

### コンポーネント

- 関数コンポーネントとHooksを使用する
- PropTypesまたはTypeScriptで型を定義する
- 1ファイル1コンポーネントの原則
- コンポーネントにはJSDocコメントを追加する

```jsx
/**
 * 商品カードコンポーネント
 * @param {Object} props
 * @param {Object} props.product - 商品情報
 * @param {Function} props.onAddToCart - カートに追加するコールバック
 */
function ProductCard({ product, onAddToCart }) {
  // ...
}
```

### CSS

- コンポーネントごとに独立したCSSファイルを作成
- BEM命名規則を推奨（Block Element Modifier）
- グローバルなスタイルは最小限に

## コミットメッセージ規約

Conventional Commitsに従います：

```
<type>: <subject>

<body>

<footer>
```

### Type

- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの動作に影響しない変更（空白、フォーマットなど）
- `refactor`: バグ修正も機能追加もしないコード変更
- `perf`: パフォーマンス改善
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツールの変更

### 例

```
feat: カテゴリフィルター機能を追加

商品一覧ページにカテゴリフィルターを追加しました。
ユーザーは特定のカテゴリの商品のみを表示できます。

Closes #123
```

## プルリクエストのプロセス

1. **ブランチを最新に保つ**: mainブランチから最新の変更を取り込む

```bash
git checkout main
git pull origin main
git checkout your-branch
git rebase main
```

2. **テストを実行**: すべてのテストが通ることを確認

```bash
npm test
```

3. **リントチェック**: コードがリント規則に従っていることを確認

```bash
npm run lint
```

4. **プルリクエストを作成**: 以下を含める
   - 変更内容の明確な説明
   - 関連するIssue番号（例: `Fixes #123`）
   - スクリーンショット（UI変更の場合）
   - テスト結果

5. **レビューに対応**: レビュアーのフィードバックに対応する

6. **マージ**: 承認後、メンテナがマージします

## プルリクエストテンプレート

プルリクエストを作成する際は、以下の情報を含めてください：

```markdown
## 概要
<!-- 変更内容の簡潔な説明 -->

## 変更の種類
- [ ] バグ修正
- [ ] 新機能
- [ ] リファクタリング
- [ ] ドキュメント更新

## 関連Issue
<!-- Closes #123 -->

## 変更内容
<!-- 詳細な変更内容を箇条書きで -->
- 
- 

## テスト
<!-- どのようなテストを実施したか -->
- [ ] ローカルで動作確認
- [ ] 既存テストが通ることを確認
- [ ] 新しいテストを追加

## スクリーンショット
<!-- UI変更がある場合 -->

## チェックリスト
- [ ] コードがリント規則に従っている
- [ ] 適切なコミットメッセージを使用している
- [ ] ドキュメントを更新した（必要な場合）
- [ ] テストを追加・更新した
```
